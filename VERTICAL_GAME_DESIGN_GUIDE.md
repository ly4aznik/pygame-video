# Vertical Competition Game Design Guide

Универсальный дизайн-код для вертикальных игровых симуляций: захват территории, гонки ботов, арены, шарики, змейки и другие короткие соревнования для Reels/Shorts.

Главная задача формата: зритель должен за одну секунду понять, кто соревнуется, что считается победой и какое событие может резко изменить матч.

## 1. Формат экрана

Базовое игровое разрешение:

```python
WIDTH, HEIGHT = 480, 960
FPS = 60
RECORD_FPS = 30
FINAL_VIDEO_SIZE = (1080, 1920)
```

Правила:

- Проектировать интерфейс сразу вертикальным.
- Важные события держать в центральных 70% экрана.
- Верх использовать для заголовка и таймера.
- Центр отдавать игровому полю.
- Низ использовать для рейтинга и статистики.
- При экспорте сохранять соотношение сторон. Нельзя растягивать изображение.

Если соотношение сторон игры и итогового видео различается:

```text
scale с force_original_aspect_ratio=decrease
затем pad до целевого разрешения
```

## 2. Компоновка

Экран делится на четыре смысловых слоя:

```text
┌────────────────────────────┐
│ Заголовок, правило, таймер │
├────────────────────────────┤
│                            │
│       Игровое поле         │
│                            │
├────────────────────────────┤
│ Соотношение владения       │
├────────────────────────────┤
│ Динамический рейтинг       │
└────────────────────────────┘
```

Рекомендуемые пропорции:

| Зона | Доля высоты |
|---|---:|
| Заголовок и таймер | 10-13% |
| Игровое поле | 38-48% |
| Полоса прогресса | 3-5% |
| Статистика и рейтинг | 32-42% |

Игровое поле должно быть самым заметным элементом, но нижний рейтинг обязан читаться без увеличения изображения.

## 3. Визуальная палитра

Использовать тёмный спокойный фон и яркие цвета участников.

```python
BG = (8, 10, 18)
PANEL = (18, 22, 35)
GRID = (38, 45, 65)
NEUTRAL_CELL = (13, 17, 29)

WHITE = (245, 247, 255)
MUTED = (170, 178, 200)
DANGER = (255, 70, 90)
WARNING = (255, 190, 70)

PLAYER_COLORS = [
    (255, 210, 75),   # yellow
    (90, 240, 150),   # green
    (255, 70, 90),    # coral
    (70, 150, 255),   # blue
    (190, 100, 255),  # purple
    (55, 225, 225),   # teal
]
```

Правила цвета:

- Один участник всегда использует один цвет во всех элементах.
- Цвет участника применяется к голове, территории, имени, проценту и уведомлениям.
- Не использовать декоративные блики внутри каждой клетки: они создают визуальный шум.
- Мёртвых участников приглушать, но не скрывать из рейтинга.
- Опасный нейтральный объект делать белым или красным.

## 4. Игровое поле

Поле должно выглядеть плотным и детализированным, но оставаться читаемым на телефоне.

Рекомендации:

- Использовать простую геометрию: прямоугольник, круг, hex-поле или компактный полигон.
- Нейтральные клетки должны быть почти фоновыми.
- Захваченные клетки окрашивать цветом владельца с прозрачностью около 65-75%.
- Границу поля рисовать ярче внутренней сетки.
- Головы участников должны быть крупнее клеток и иметь подписи.
- Стартовые позиции размещать на 3-5 клеток от края, а не прямо на границе.

Для плотного поля в окне `480x960` подходит сетка около `48x41` с клеткой `9px`.

## 5. Заголовок и таймер

Верхний блок должен объяснять конфликт короткой фразой:

```text
PICK YOUR SNAKE
WHITE BALL ERASES TERRITORY
```

Правила:

- Главный заголовок: 2-5 слов, uppercase, высокий контраст.
- Подзаголовок: одно короткое правило или угроза.
- Обычный таймер небольшой и расположен по центру.
- За 15 секунд до конца таймер увеличивается и пульсирует.
- Последние секунды сопровождаются отдельным звуком.

## 6. Полоса владения

Между игровым полем и статистикой размещается единая полоса контроля.

Требования:

- Каждый сегмент окрашен цветом участника.
- Ширина сегмента равна доле владения относительно остальных.
- Полоса обновляется непрерывно.
- Мёртвые участники отображаются приглушённым цветом.
- Подпись должна быть короткой: `LIVE TERRITORY`, `CONTROL` или `OWNERSHIP`.

Эта полоса позволяет понять лидера без чтения цифр.

## 7. Динамический рейтинг

Нижний блок показывает не просто список участников, а турнирную таблицу.

Рекомендуемые столбцы:

```text
PLAYER / STRATEGY | BLOCKS | CONTROL | STATUS
```

Где:

- `PLAYER / STRATEGY` — имя и короткое описание поведения.
- `BLOCKS` — число пройденных клеток или действий за матч.
- `CONTROL` — процент владения относительно всех участников.
- `STATUS` — `small`, `big`, `dead` или другой простой статус.

Сортировка:

```python
sorted(
    players,
    key=lambda player: (
        -player.owned_cells,
        -player.blocks_travelled,
        player.name,
    ),
)
```

Правила:

- Участник с наибольшим процентом всегда находится сверху.
- Порядок обновляется раз в игровую секунду, а не каждый кадр.
- Это предотвращает постоянное дрожание строк.
- При равенстве владения выше стоит участник с большей активностью.
- Перестановка мест должна быть заметной, но не отвлекать от игрового поля.

## 8. Уведомления о событиях

В правом верхнем углу поля показываются компактные сообщения как в мессенджере.

Примеры:

```text
MIA BECAME BIG
LEO CAPTURED 12 CELLS
SAM WAS ERASED
```

Правила:

- Новое сообщение появляется сверху.
- Старые сообщения сдвигаются вниз.
- Сообщение живёт около 5 секунд.
- Одновременно показывать не больше 3-4 сообщений.
- Использовать цвет участника для рамки, имени и аватара.
- Не перекрывать большую часть игрового поля.
- Частые однотипные события ограничивать cooldown.

## 9. Финальный экран

Финал должен однозначно завершать историю матча.

Показывать:

```text
WINNER
PLAYER NAME
STRATEGY
Blocks N  Control N%
REASON
```

Правила:

- Победителем не может стать участник со статусом `dead`, если остались живые.
- Цвет победителя используется для рамки и glow.
- Фон затемняется.
- Центральная карточка крупная, но не касается краёв.
- Причина завершения должна быть короткой: `TIME LIMIT`, `LAST ALIVE`.

## 10. Звуковой дизайн

Звуки должны быть нарочито мультяшными и легко различаться.

Рекомендуемая палитра:

| Событие | Характер звука |
|---|---|
| Обычный захват | короткий `pop` |
| Захват области | пружинистый `boing` |
| Бонус | яркий восходящий `sparkle` |
| Уведомление | короткий звонкий акцент |
| Стирание/разрушение | комичный `splat` |
| Смерть | громкий грубый `bonk` |
| Таймер | короткий мультяшный щелчок |
| Последние секунды | более громкий `boing` |
| Победа | восходящий свистящий эффект |

Фоновая музыка:

- Игривый короткий loop.
- Тише событийных звуков.
- Не должна маскировать смерть, таймер или бонус.
- Во время громкого события музыку можно кратко приглушать.

Запись должна захватывать звуки без ускорения или изменения высоты тона.

## 11. Движение и читаемость

Каждый участник должен иметь стратегию, которую зритель может заметить визуально.

Примеры:

- Движется кругами, но стремится к новым клеткам.
- Держится левой стороны.
- Идёт зигзагами.
- Строит спираль.
- Ищет самые далёкие свободные клетки.
- Следует за опасным объектом, но уклоняется от столкновения.

Общий приоритет действий:

```text
1. Избежать немедленной смерти.
2. Пойти к доступному бонусу.
3. Захватить достижимую свободную область.
4. Следовать индивидуальной стратегии.
5. Двигаться внутри своей зоны только при отсутствии расширения.
```

Стратегия не должна заставлять участника бесконечно двигаться внутри собственной территории.

## 12. Темп матча

Короткая симуляция должна постоянно давать новые события.

Рекомендуемый ритм:

- Первое заметное событие: в первые 1-3 секунды.
- Первый бонус: примерно на 5-й секунде.
- Значимое событие: каждые 2-5 секунд.
- Обратный отсчёт: последние 15 секунд.
- Финальный экран: 1-2 секунды после завершения.

Опасный объект может ускоряться после каждого отскока, чтобы напряжение росло естественно.

## 13. Запись видео

Запись должна идти с обычной скоростью игры.

Правила:

- Не замедлять игровое время при включённой записи.
- Не ускорять итоговое видео.
- Не изменять длительность звуков.
- Записывать игровой звук вместе с видео.
- Сохранять логи событий рядом с записью.
- При изменении разрешения сохранять исходные пропорции.

Рекомендуемый ffmpeg-фильтр:

```text
scale=TARGET_W:TARGET_H:
force_original_aspect_ratio=decrease:
force_divisible_by=2,
pad=TARGET_W:TARGET_H:(ow-iw)/2:(oh-ih)/2:color=black,
setsar=1
```

## 14. Чек-лист переноса на другую игру

Перед применением дизайна к новой игре проверить:

- [ ] Конфликт понятен за одну секунду.
- [ ] У каждого участника уникальный цвет и короткое имя.
- [ ] Главная угроза визуально отличается от участников.
- [ ] Поле занимает центральную часть экрана.
- [ ] Есть таймер и усиленный финальный отсчёт.
- [ ] Есть полоса общего прогресса или владения.
- [ ] Нижняя таблица сортируется по главному показателю.
- [ ] Таблица обновляет места не чаще раза в секунду.
- [ ] Есть компактные уведомления о важных событиях.
- [ ] Статусы используют короткие понятные слова.
- [ ] Мёртвые участники не могут победить при наличии живых.
- [ ] Звуки событий различимы и громче фоновой музыки.
- [ ] Игра читается без звука.
- [ ] Запись не меняет скорость игры и звука.
- [ ] Экспорт не растягивает изображение.

## 15. Минимальный шаблон

Для быстрой адаптации новой игры достаточно реализовать:

```text
1. Верх: заголовок + правило + таймер.
2. Центр: поле с яркими участниками и одной угрозой.
3. Под полем: полоса прогресса.
4. Низ: динамический рейтинг.
5. Правый верх поля: события.
6. Последние 15 секунд: пульсирующий таймер.
7. Финал: крупная карточка победителя.
8. Мультяшные звуки для каждого типа события.
```

Если все восемь пунктов работают, игра уже соответствует этому визуальному формату.
