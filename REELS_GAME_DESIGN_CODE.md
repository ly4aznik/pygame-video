# Design Code для Reels-игр и визуальных симуляций

Этот документ — инструкция для Codex/LLM-агента, который генерирует короткие игровые симуляции для Instagram Reels: прыгающие шарики, захват территории, AI-битвы, змейки, hex-карты, mini game challenges и другие залипательные форматы.

Главная цель: **каждая игра должна быть понятна за 0.5–1 секунду, визуально читаться без звука и давать зрителю повод досмотреть до конца**.

---

## 1. Базовая идея формата

Такие ролики работают не как “игра”, а как **мини-соревнование с мгновенно понятным правилом**.

Плохая формулировка:

> Красивая физическая симуляция шариков.

Хорошая формулировка:

> Один красный шар против двадцати синих. Кто выживет?

Плохая формулировка:

> Змейки бегают по полю.

Хорошая формулировка:

> Шесть змей захватывают hex-карту, но белый eraser-ball стирает территорию.

Почти каждая симуляция должна отвечать на один из вопросов:

- Кто победит?
- Кто выживет?
- Кто захватит больше территории?
- Какой цвет останется последним?
- Сможет ли слабый победить сильного?
- Успеет ли объект добраться до цели?
- Сможет ли зритель уследить за правильным объектом?

---

## 2. Технический формат Reels

Все игры должны проектироваться сразу под вертикальный short-form.

```python
CANVAS_W = 1080
CANVAS_H = 1920
FPS = 30
DURATION_SECONDS = 15  # обычно 12–25 секунд
TOTAL_FRAMES = FPS * DURATION_SECONDS
```

Рекомендуемые параметры:

| Параметр | Значение |
|---|---|
| Разрешение | `1080x1920` |
| Соотношение сторон | `9:16` |
| FPS | `30`, иногда `60`, если много быстрых объектов |
| Длительность | `12–25 секунд` |
| Экспорт | `mp4`, `H.264`, `yuv420p` |
| Обложка | `1080x1920`, но важное держать в центральной зоне |
| Основной текст | 2–5 слов |
| Хэштеги в caption | максимум 5, но это не часть игры |

## 3. Safe zones

У Instagram есть интерфейсные элементы: кнопки справа, описание снизу, верхние элементы навигации. Поэтому нельзя ставить важный текст и победителя в самые края.

Используй такие зоны:

```python
SAFE_MARGIN_X = 90
SAFE_TOP = 180
SAFE_BOTTOM = 360
GRID_SAFE_TOP = 240
GRID_SAFE_BOTTOM = 240

SAFE_RECT = pygame.Rect(
    SAFE_MARGIN_X,
    SAFE_TOP,
    CANVAS_W - SAFE_MARGIN_X * 2,
    CANVAS_H - SAFE_TOP - SAFE_BOTTOM,
)

GRID_SAFE_RECT = pygame.Rect(
    0,
    GRID_SAFE_TOP,
    CANVAS_W,
    CANVAS_H - GRID_SAFE_TOP - GRID_SAFE_BOTTOM,
)
```

Правила:

- Главный конфликт должен быть виден в центре.
- Заголовок обложки — в верхней трети, но не у самого края.
- Таймер и счёт лучше ставить в верхнюю центральную часть.
- Победителя показывать в центре или чуть выше центра.
- Не ставить важные элементы в правую нижнюю часть: там часто интерфейс Instagram.
- Не ставить мелкий текст внизу.

## 4. Главный принцип дизайна

Каждый ролик строится вокруг формулы:

```text
Один понятный конфликт + яркий визуальный контраст + ожидание развязки
```

Примеры сильных конфликтов:

```text
1 giant ball vs 20 small balls
Red vs Blue territory
Snake bots vs eraser ball
Fast bot vs smart bot
One survivor in shrinking arena
Follow the red ball
```

Если зрителю нужно больше 1 секунды, чтобы понять, что происходит, дизайн надо упростить.

---

## 5. Визуальная иерархия

На экране всегда должны быть уровни важности.

### Уровень 1 — главный конфликт

Это самое крупное и контрастное:

- главный шар-босс;
- лидирующая змейка;
- центр карты;
- опасная зона;
- eraser-ball;
- победитель.

### Уровень 2 — вторичные участники

Это остальные боты, шарики, фракции, клетки.

### Уровень 3 — фон и декоративные эффекты

Фон, частицы, сетка, glow, trail, вспышки.

Фон не должен спорить с объектами. Если объекты яркие, фон должен быть тёмным и спокойным.

## 6. Палитра

Используй 3–6 основных цветов. Не делай радугу без причины.

### Базовая тёмная палитра

```python
COLORS = {
    "bg": (8, 10, 18),
    "panel": (18, 22, 35),
    "grid": (38, 45, 65),
    "text": (245, 247, 255),
    "muted_text": (170, 178, 200),
    "danger": (255, 70, 90),
    "warning": (255, 190, 70),
    "white": (255, 255, 255),
}
```

### Цвета фракций

```python
FACTION_COLORS = [
    (255, 70, 90),    # red / danger
    (70, 150, 255),   # blue
    (90, 240, 150),   # green
    (190, 100, 255),  # purple
    (255, 210, 75),   # yellow
    (255, 120, 220),  # pink
]
```

Правила цвета:

- У каждой фракции должен быть один узнаваемый цвет.
- Победитель в финале подсвечивается сильнее остальных.
- Опасный объект почти всегда белый, красный или кислотный.
- Если фон светлый, объекты должны быть тёмными или с обводкой.
- Не использовать близкие цвета для соперников: например, синий и голубой хуже, чем синий и оранжевый.

## 7. Типографика

Текст должен читаться на телефоне за долю секунды.

### Текст на обложке

Длина: **2–5 слов**.

Хорошие варианты:

```text
WHO WINS?
1 VS MANY
DON'T BLINK
PICK A COLOR
DEADLY LAKE
LAST 3 SECONDS
RED VS BLUE
IMPOSSIBLE COMEBACK
```

Плохие варианты:

```text
This is a simulation where multiple agents are competing for territory
A very interesting physics experiment with balls
Watch this amazing satisfying animation
```

### Размеры текста

```python
FONT_HOOK_SIZE = 92
FONT_BIG_SIZE = 120
FONT_MEDIUM_SIZE = 56
FONT_SMALL_SIZE = 34
```

Правила:

- Текст должен иметь обводку или тень.
- Не использовать больше двух шрифтов.
- Лучше один жирный шрифт для всего проекта.
- На обложке максимум 1 главный текстовый блок.
- В ролике текст должен исчезать через 1–2 секунды, чтобы не мешать симуляции.

Пример функции текста с обводкой:

```python
def draw_text_with_outline(surface, text, font, center, fill, outline=(0, 0, 0), width=4):
    x, y = center
    for dx in range(-width, width + 1):
        for dy in range(-width, width + 1):
            if dx * dx + dy * dy <= width * width:
                img = font.render(text, True, outline)
                rect = img.get_rect(center=(x + dx, y + dy))
                surface.blit(img, rect)
    img = font.render(text, True, fill)
    rect = img.get_rect(center=center)
    surface.blit(img, rect)
```

## 8. Композиция обложки

Обложка должна продавать не “красоту”, а **интригу**.

### Шаблон 1: 1 VS MANY

Использовать для прыгающих шариков, боссов, swarm-сцен.

Композиция:

```text
верх: крупный текст “1 VS MANY”
центр: один огромный шар
вокруг: 10–30 маленьких шаров
фон: тёмная арена
низ: без важного текста
```

Ключ: зритель мгновенно видит дисбаланс.

### Шаблон 2: WHO WINS?

Использовать для дуэлей.

Композиция:

```text
верх: “WHO WINS?”
левая часть: красная сторона
правая часть: синяя/зелёная сторона
центр: столкновение, граница или взрыв
низ: таймер или маленький счётчик, если нужен
```

Ключ: зритель выбирает сторону.

### Шаблон 3: DEADLY LAKE / VOID CORE

Использовать для territory capture.

Композиция:

```text
верх: “DEADLY LAKE”
центр: опасная зона
вокруг: фракции и следы территории
фон: hex-grid или карта
```

Ключ: на карте есть необычная угроза, а не просто закрашивание клеток.

### Шаблон 4: DON’T BLINK

Использовать для игр на внимание.

Композиция:

```text
верх: “DON'T BLINK”
центр: один красный target-object
вокруг: много похожих объектов
низ: без мелких пояснений
```

Ключ: зритель понимает, что его проверяют.

## 9. Начало ролика

Нельзя начинать с меню, заставки, логотипа, медленного появления поля.

Первые 0.5 секунды должны показывать:

- движение;
- конфликт;
- цветовые стороны;
- угрозу или цель;
- короткий hook-текст.

Пример структуры:

```text
0.0–0.5 сек: уже идёт столкновение / захват / погоня
0.5–1.5 сек: появляется hook “WHO WINS?”
1.5–12 сек: основная симуляция
последние 2–3 сек: ускорение, драматичная развязка
последние 0.5 сек: freeze-frame победителя
```

---

## 10. Движение и анимация

Движение должно быть залипательным, но читаемым.

Правила:

- Объекты не должны быть слишком мелкими.
- Скорость должна быть высокой, но не хаотичной.
- Траектории должны оставлять следы, если это помогает понять игру.
- Столкновения должны давать визуальную обратную связь: вспышка, частицы, shake, звук при наличии аудио.
- Ключевые события надо подсвечивать.

### Эффекты, которые стоит использовать

```text
glow вокруг лидера
trail за быстрым объектом
частицы при столкновении
короткий camera shake при важном ударе
пульсация danger-zone
замедление или freeze-frame на победителе
```

Не использовать:

```text
слишком много blur
мелкие частицы поверх всего экрана
длинные fade-in/fade-out
сложные UI-панели
много текста одновременно
```

---

## 11. UI внутри ролика

UI должен быть минимальным.

Разрешённые элементы:

- таймер;
- счёт фракций;
- проценты территории;
- имя текущего лидера;
- короткий hook;
- финальный winner banner.

Пример UI:

```text
верхняя зона: TIMER 00:12
центр: игровая сцена
нижняя зона: пусто или очень лёгкий score strip
финал: WINNER: GREEN
```

Лучше не делать постоянную таблицу результатов. Она занимает внимание и снижает залипательность.

## 12. Архитектура проекта

Codex должен создавать проект с понятной структурой.

```text
project/
  main.py              # CLI, запуск preview/render
  config.py            # размеры, FPS, цвета, параметры
  simulation.py        # логика мира
  entities.py          # боты, шарики, агенты
  renderer.py          # отрисовка сцены
  effects.py           # частицы, glow, shake, trails
  covers.py            # генерация обложек
  export.py            # экспорт кадров и ffmpeg-команды
  captions.py          # варианты caption и cover text
  README.md            # запуск и параметры
  frames/              # экспорт кадров
  covers/              # cover_a.png, cover_b.png
  output/              # итоговые mp4
```

Если проект маленький, можно объединить файлы, но логика должна оставаться разделённой:

```text
Simulation отвечает за правила.
Renderer отвечает за картинку.
Export отвечает за видео.
Covers отвечает за обложки.
```

## 13. Конфиг игры

В каждой игре должен быть единый объект настроек.

```python
from dataclasses import dataclass

@dataclass
class GameConfig:
    width: int = 1080
    height: int = 1920
    fps: int = 30
    duration: int = 18
    seed: int = 42
    export_frames: bool = False
    show_overlay: bool = True
    generate_covers: bool = True
    safe_margin_x: int = 90
    safe_top: int = 180
    safe_bottom: int = 360
```

Все случайные элементы должны зависеть от `seed`, чтобы можно было повторить удачный ролик.

Пример:


Минимальные аргументы:

```text
--preview

--seed
--duration
--fps
--no-overlay
--covers-only

```

## 15. Экспорт кадров и видео

Код должен уметь сохранять кадры:

```python
pygame.image.save(surface, f"frames/frame_{frame_index:05d}.png")
```

Команда ffmpeg:

```bash
ffmpeg -framerate 30 -i frames/frame_%05d.png \
  -vf "fps=30,format=yuv420p" \
  -c:v libx264 -profile:v high -pix_fmt yuv420p \
  -preset slow -crf 18 -movflags +faststart \
  -an output/reel.mp4
```

С музыкой:

```bash
ffmpeg -framerate 30 -i frames/frame_%05d.png -i audio/music.wav \
  -vf "fps=30,format=yuv420p" \
  -c:v libx264 -profile:v high -pix_fmt yuv420p \
  -preset slow -crf 18 -movflags +faststart \
  -c:a aac -b:a 192k -ar 48000 -shortest output/reel_with_audio.mp4
```

## 16. Генерация обложек

Каждая игра должна генерировать минимум две обложки.

```text
cover_a.png — с текстовым hook
cover_b.png — без текста, но с сильным визуальным конфликтом
```

Обложка A:

```text
крупный текст
максимально понятный конфликт
фон затемнён
объекты крупные
```

Обложка B:

```text
нет текста
самый драматичный кадр
видна асимметрия или почти победа
```

Функция генерации:

```python
def render_cover(surface, title: str, subtitle: str | None = None):
    # 1. затемнить фон
    # 2. увеличить главные объекты
    # 3. добавить glow/outline
    # 4. нарисовать title в верхней safe-zone
    # 5. сохранить 1080x1920
    pass
```

## 17. Механики, которые подходят под Reels

### 17.1 Bouncing Ball Arena

Формула:

```text
один сильный объект против многих слабых
```

Обязательные элементы:

- главный шар крупнее остальных;
- понятная арена;
- много столкновений;
- визуальный счёт здоровья или размера;
- финальный survivor.

Hook:

```text
1 VS MANY
WHO SURVIVES?
BIG RED VS EVERYONE
```

### 17.2 Territory Capture

Формула:

```text
цветные агенты закрашивают поле, но карта имеет опасность
```

Обязательные элементы:

- цветные следы;
- проценты территории;
- hazard: lake, void, eraser, vortex;
- финальный подсчёт.

Hook:

```text
WHO TAKES THE MAP?
DEADLY LAKE
PICK YOUR COLOR
```

### 17.3 Hex Snake Battle

Формула:

```text
змейки с разными стратегиями соревнуются за клетки
```

Обязательные элементы:

- hex-grid;
- змейки разных цветов;
- понятные стратегии: hunter, greedy, random, cautious;
- eraser или shrinking zone для хаоса;
- финальный leaderboard.

Hook:

```text
PICK YOUR SNAKE
WHO CONTROLS THE BOARD?
SMART VS FAST
```

### 17.4 Follow The Ball

Формула:

```text
зритель должен уследить за объектом
```

Обязательные элементы:

- один target-object;
- похожие distracting objects;
- короткий countdown;
- финальный reveal;
- CTA на комментарий.

Hook:

```text
DON'T BLINK
FOLLOW RED
COMMENT THE NUMBER
```

## 18. Поведение ботов

У ботов должны быть простые и узнаваемые стратегии.

```python
class Strategy:
    RANDOM = "random"
    GREEDY = "greedy"
    HUNTER = "hunter"
    CAUTIOUS = "cautious"
    CENTER_CONTROL = "center_control"
    EDGE_RUNNER = "edge_runner"
```

Каждая стратегия должна визуально отличаться:

| Стратегия | Поведение | Визуальное ощущение |
|---|---|---|
| Random | хаотично выбирает направление | непредсказуемость |
| Greedy | идёт к ближайшей выгодной клетке | быстрый рост |
| Hunter | атакует лидера | драматичные столкновения |
| Cautious | избегает опасности | стабильность |
| Center Control | держит центр | доминирование |
| Edge Runner | бегает по краю | неожиданные захваты |

Не надо делать сложный ИИ. В Reels важнее, чтобы поведение **читалось глазами**.

---

## 19. Камера

Обычно камера фиксированная. Но можно использовать лёгкие эффекты:

```text
camera shake при взрыве
smooth zoom на финального победителя
короткий zoom-out в конце, чтобы показать всё поле
```

Не использовать:

```text
постоянное движение камеры
сильное приближение без причины
вращение поля
```

---

## 20. Финал ролика

Финал должен быть чётким. Зритель должен понять результат без caption.

Финальный экран:

```text
WINNER: GREEN
68% CONTROLLED
RED ELIMINATED
```

Финал должен длиться 0.5–1.2 секунды.

Хороший финал:

- freeze-frame;
- подсветка победителя;
- короткий текст;
- частицы или glow;
- видно итоговое поле.

Плохой финал:

- ролик резко обрывается;
- непонятно, кто победил;
- счёт мелкий;
- последний кадр пустой.

---

## 21. Loop-friendly дизайн

Ролик должен нормально смотреться при повторе.

Приёмы:

- начинать сразу с действия;
- финал не делать слишком длинным;
- последний кадр можно визуально связать с первым;
- не использовать длинные титры;
- можно делать “almost seamless loop”, когда после финала снова начинается похожая арена.

---

## 22. Что Codex обязан проверять перед завершением

Перед финальным ответом Codex должен проверить:

```text
[ ] Игра запускается без ошибок.
[ ] Есть preview-режим.
[ ] Есть render-режим.
[ ] Размер canvas 1080x1920.
[ ] Важные элементы не находятся у краёв.
[ ] Первые 0.5 секунды уже содержат движение.
[ ] Есть понятный конфликт.
[ ] Есть финальная развязка.
[ ] Есть минимум две обложки.
[ ] Есть ffmpeg-команда.
[ ] Есть seed для повторяемости.
[ ] Нет чужих ассетов и водяных знаков.
[ ] Цвета читаются на телефоне.
[ ] Текст на обложке не длиннее 5 слов.
```

## 23. Запрещённые решения

Не делать:

```text
- горизонтальный canvas;
- длинное меню в начале;
- мелкие объекты без zoom;
- серый фон и слабый контраст;
- 10 разных шрифтов;
- мелкий текст внизу;
- слишком сложные правила;
- много параметров, которые зритель не видит;
- скучный финал без winner reveal;
```

## 25. Быстрый чеклист качества ролика

Перед публикацией ответь на вопросы:

1. Понятно ли за 1 секунду, кто с кем соревнуется?
2. Есть ли причина досмотреть до конца?
3. Можно ли выбрать сторону?
4. Видно ли победителя?
5. Есть ли один главный визуальный конфликт?
6. Не мешает ли UI Instagram важным элементам?
7. Работает ли обложка как отдельный постер?
8. Есть ли 2–3 варианта caption для теста?
9. Можно ли сделать ещё 10 роликов по этой же формуле?
10. Хочется ли отправить это другу?

Если хотя бы на 3 вопроса ответ “нет”, дизайн надо упростить или усилить конфликт.

## 26. Главная формула

```text
Простое правило + яркий конфликт + понятная ставка + сильный финал = хороший Reels-формат
```

Не пытайся делать “сложную игру”. Делай **понятный эксперимент**, в котором зритель за секунду выбирает сторону и ждёт развязку.
